# パフォーマンス技術仕様書作成ガイドライン

## 🎯 概要

このガイドラインは、**実サービスのパフォーマンス技術仕様書を作成するための標準的な手法**を定義します。
具体的なサービス要件に基づいて、測定可能で実現可能なパフォーマンス仕様を文書化する方法を示します。

### 対象ドキュメント
このガイドラインに従って作成される技術仕様書：
- `performance-requirements-spec.md` - パフォーマンス要件仕様書
- `performance-testing-plan.md` - パフォーマンステスト計画書
- `performance-monitoring-spec.md` - パフォーマンス監視仕様書

## 1. パフォーマンス要件の定義方法

### 🎯 要件定義のアプローチ

パフォーマンス技術仕様書を作成するには、以下の段階的アプローチを取ります：

#### 1.1 ビジネス要件からの導出
```
ビジネス要件 → ユーザー体験要件 → 技術パフォーマンス要件
```

**例：ECサイトの場合**
- **ビジネス要件**: 「ユーザーの離脱率を下げ、コンバージョン率を向上させる」
- **ユーザー体験要件**: 「ページ読み込みが早く、操作に即座に反応する」
- **技術パフォーマンス要件**: 「LCP < 2.5s、FID < 100ms、API応答時間 < 300ms」

#### 1.2 利用シーナリオ別の要件整理

サービスの主要利用パターンを特定し、それぞれに適したパフォーマンス要件を定義：

| 利用シーナリオ | 重要度 | 主要ユーザー | パフォーマンス要件の焦点 |
|---------------|--------|-------------|----------------------|
| **初回訪問** | 最高 | 新規ユーザー | 初期表示速度（LCP、TTFB） |
| **ログイン後操作** | 高 | 既存ユーザー | インタラクション応答（FID、API応答） |
| **管理機能** | 中 | 管理者 | データ処理性能（複雑クエリ、バッチ処理） |
| **モバイル利用** | 高 | モバイルユーザー | ネットワーク効率（データ転送量、オフライン対応） |

#### 1.3 競合分析による目標設定

競合サービスのパフォーマンスを調査し、目標値を設定：

```markdown
## 競合分析結果（例）

| サービス | LCP | FID | API応答時間 | 備考 |
|---------|-----|-----|------------|------|
| 競合A | 1.8s | 45ms | 250ms | 業界リーダー |
| 競合B | 2.3s | 80ms | 400ms | 価格重視 |
| 競合C | 1.2s | 30ms | 180ms | プレミアム |
| **我々の目標** | **< 1.5s** | **< 50ms** | **< 200ms** | **競合上位を目指す** |
```

## 2. 技術仕様書の構成

### 📋 標準的な仕様書構成

パフォーマンス技術仕様書は以下の構成で作成することを推奨します：

#### 2.1 基本構成要素

```markdown
# [サービス名] パフォーマンス要件仕様書

## 1. 概要・目的
- サービス概要
- パフォーマンス要件定義の背景・目的
- ビジネス影響の説明

## 2. スコープ・対象範囲
- 対象となるシステム範囲
- 除外項目の明確化
- 対象ユーザー・利用環境の定義

## 3. パフォーマンス要件
### 3.1 機能性能要件
### 3.2 非機能性能要件
### 3.3 可用性要件

## 4. 測定・監視要件
### 4.1 測定指標
### 4.2 測定環境・条件
### 4.3 監視・アラート要件

## 5. テスト要件
### 5.1 性能テスト要件
### 5.2 負荷テスト要件
### 5.3 受け入れ基準

## 6. 制約・前提条件
### 6.1 技術的制約
### 6.2 運用制約
### 6.3 コスト制約

## 7. リスク・対応策
### 7.1 パフォーマンスリスク
### 7.2 軽減・対応策
```

#### 2.2 各セクションの記述ガイドライン

**概要・目的**
- なぜそのパフォーマンス要件が必要なのかを明確に記述
- ビジネス価値への影響を定量的に示す

**スコープ・対象範囲**
- 測定対象の境界を明確にする
- 「何を測定しないか」も明記する

**パフォーマンス要件**
- 具体的な数値目標を設定
- 測定条件・環境を併記
- 目標値・許容値・限界値を区別

#### 2.3 要件記述の品質基準

**良い要件記述の例**：
```markdown
### ユーザー認証API応答時間
- **目標値**: 95%のリクエストが200ms以内
- **許容値**: 99%のリクエストが500ms以内
- **測定条件**: 
  - 同時接続ユーザー数: 1,000人
  - ネットワーク環境: 4G回線相当
  - 測定期間: 平日営業時間帯（9:00-18:00）
- **ビジネス影響**: 応答時間500ms超過でユーザー離脱率15%増加
```

**悪い要件記述の例**：
```markdown
### 認証APIは高速であること
- 早いレスポンスが必要
- ユーザー体験に影響するため
```

## 3. 測定指標の選定基準

### 📊 指標選定の原則

パフォーマンス測定指標を選定する際の基準：

#### 3.1 ビジネス価値との関連性

指標がビジネス成果に与える影響を明確にする：

| 指標カテゴリ | ビジネス価値 | 主要指標例 | 影響度測定方法 |
|-------------|-------------|------------|---------------|
| **ユーザー体験** | コンバージョン率、顧客満足度 | LCP, FID, CLS | A/Bテスト、コホート分析 |
| **運用効率** | 運用コスト、開発生産性 | デプロイ時間、MTTR | 工数・コスト計測 |
| **システム安定性** | 信頼性、ブランド価値 | 可用性、エラー率 | 障害影響度分析 |

#### 3.2 測定可能性・実現可能性

技術的に測定可能で、改善アクションが取れる指標を選択：

**測定可能性チェックリスト**：
- [ ] 自動測定が可能である
- [ ] 測定データが継続的に取得できる
- [ ] 測定結果の解釈が明確である
- [ ] 測定コストが合理的である

**実現可能性チェックリスト**：
- [ ] 現在の技術スタックで測定可能
- [ ] 必要なツール・インフラが利用可能
- [ ] 測定精度が要求水準を満たす
- [ ] チーム内で測定・分析スキルがある

#### 3.3 指標の階層化

パフォーマンス指標を3層に分けて管理：

```markdown
## 指標階層の例

### レベル1: ビジネス指標（経営層向け）
- 月間アクティブユーザー数
- コンバージョン率
- 顧客満足度スコア
- システム稼働率

### レベル2: サービス指標（サービス責任者向け）
- ページ読み込み時間（P95）
- API応答時間（P95）
- エラー率
- 同時接続ユーザー数

### レベル3: 技術指標（開発・運用チーム向け）
- CPU使用率
- メモリ使用率
- データベースクエリ実行時間
- キャッシュヒット率
```

#### 3.4 Core Web Vitals の適用指針

Google Core Web Vitals を自社サービスに適用する際の考慮事項：

**LCP (Largest Contentful Paint)**
- **適用場面**: コンテンツ重視のページ（ブログ、ECサイト）
- **目標設定**: ユーザーの期待値と競合分析に基づく
- **除外ケース**: SPA の動的コンテンツページ

**FID (First Input Delay)**
- **適用場面**: インタラクション重視のページ（ダッシュボード、ツール）
- **目標設定**: ユーザーアクションの緊急度に応じて設定
- **除外ケース**: 静的コンテンツページ

**CLS (Cumulative Layout Shift)**
- **適用場面**: 全ページ（特に広告、動的コンテンツあり）
- **目標設定**: 0.1以下を基本とし、コンテンツ特性で調整

## 4. パフォーマンス目標値の設定手法

### 🎯 目標値設定の段階的アプローチ

パフォーマンス目標値は以下の段階で設定します：

#### 4.1 ベースライン測定

現状のパフォーマンスを正確に把握：

```markdown
## ベースライン測定計画（例）

### 測定対象期間
- 測定期間: 4週間（営業日のみ）
- 測定時間帯: 9:00-18:00（ピーク時間を含む）
- 測定頻度: 15分間隔

### 測定環境
- 地理的位置: 東京、大阪、札幌（主要ユーザー地域）
- ネットワーク: 4G、光回線、WiFi
- デバイス: Desktop（Chrome, Safari）、Mobile（iOS Safari, Android Chrome）

### 測定結果記録フォーマット
| 指標 | 平均値 | P50 | P90 | P95 | P99 | 最大値 |
|------|--------|-----|-----|-----|-----|--------|
| LCP | - | - | - | - | - | - |
| FID | - | - | - | - | - | - |
| API応答時間 | - | - | - | - | - | - |
```

#### 4.2 目標値の3段階設定

現実的で段階的な目標設定を行う：

| 段階 | 期間 | 目標水準 | 設定基準 |
|------|------|----------|----------|
| **短期目標** | 3ヶ月 | 現状から20%改善 | 技術的に確実に達成可能 |
| **中期目標** | 12ヶ月 | 業界標準レベル | 競合分析に基づく |
| **長期目標** | 24ヶ月 | 業界リーダーレベル | 理想的なユーザー体験 |

#### 4.3 目標値設定の具体例

**ECサイトの場合**：

```markdown
## 商品一覧ページ パフォーマンス目標

### 現状（ベースライン）
- LCP: P95 = 3.2秒
- FID: P95 = 150ms
- 直帰率: 45%

### 短期目標（3ヶ月後）
- LCP: P95 < 2.5秒（22%改善）
- FID: P95 < 120ms（20%改善）
- 直帰率: < 40%（5ポイント改善）

### 中期目標（12ヶ月後）
- LCP: P95 < 2.0秒（業界標準）
- FID: P95 < 100ms（業界標準）
- 直帰率: < 35%（10ポイント改善）

### 長期目標（24ヶ月後）
- LCP: P95 < 1.5秒（業界上位）
- FID: P95 < 50ms（業界上位）
- 直帰率: < 30%（15ポイント改善）
```

#### 4.4 ビジネス影響を考慮した優先度付け

パフォーマンス改善の優先順位を決定：

| 改善項目 | ビジネス影響度 | 技術的難易度 | 優先度 | 実施タイミング |
|----------|---------------|-------------|--------|---------------|
| **商品検索API高速化** | 高（売上直結） | 中 | **最高** | 短期目標 |
| **画像読み込み最適化** | 中（UX改善） | 低 | 高 | 短期目標 |
| **管理画面最適化** | 低（内部効率） | 高 | 低 | 長期目標 |

#### 4.5 許容値・警告値の設定

目標値に加えて、運用上の判断基準を設定：

```markdown
## アラートレベル定義

### 正常範囲（緑）
- LCP P95 < 2.0秒
- API応答時間 P95 < 300ms
- エラー率 < 0.1%

### 注意範囲（黄）
- LCP P95: 2.0-2.5秒
- API応答時間 P95: 300-500ms
- エラー率: 0.1-1.0%

### 警告範囲（赤）
- LCP P95 > 2.5秒
- API応答時間 P95 > 500ms
- エラー率 > 1.0%
```

## 5. 制約条件の文書化

### 🚧 制約条件の分類と記述方法

パフォーマンス要件に影響する制約条件を体系的に文書化：

#### 5.1 技術的制約

システムアーキテクチャ・技術選択による制約：

```markdown
## 技術的制約事項

### アーキテクチャ制約
- **サーバーサイド**: Vercel Edge Runtime（実行時間制限30秒）
- **データベース**: PostgreSQL（接続プール上限100）
- **CDN**: Vercel CDN（地理的分散制限）
- **外部API**: レート制限（1,000req/hour）

### 技術スタック制約
- **Next.js**: App Router使用（一部機能制限）
- **Hono**: Edge Runtime対応ライブラリのみ使用可能
- **メモリ制限**: Vercel Pro（1GB上限）

### 既存システム連携制約
- **レガシーシステム**: 応答時間1-3秒（改善不可）
- **外部決済API**: 応答時間5-10秒（制御不可）
- **画像変換サービス**: 処理時間2-5秒
```

#### 5.2 運用制約

運用面での制約条件：

```markdown
## 運用制約事項

### 人的リソース制約
- **開発チーム**: 3名（フルスタック）
- **運用担当**: 1名（兼任）
- **パフォーマンス専門知識**: 限定的

### 運用時間制約
- **メンテナンス時間**: 毎週日曜 2:00-4:00
- **デプロイ可能時間**: 平日 10:00-16:00
- **障害対応**: 営業時間内のみ

### 監視・測定制約
- **監視ツール**: 無料プランの制限（保存期間30日）
- **ログ保存**: 7日間のみ
- **テスト環境**: 本番環境と構成差異あり
```

#### 5.3 コスト制約

予算・コストによる制約：

```markdown
## コスト制約事項

### インフラコスト制約
- **クラウド費用**: 月額$200以下
- **監視ツール**: 月額$50以下
- **CDN**: 転送量制限あり

### ツール・ライセンス制約
- **パフォーマンス測定ツール**: オープンソース限定
- **負荷テストツール**: 無料プラン範囲内
- **APMツール**: 基本プランのみ

### 改善投資制約
- **パフォーマンス改善**: 四半期予算$1,000
- **新規ツール導入**: 年間予算$5,000
```

#### 5.4 ビジネス制約

事業要件による制約：

```markdown
## ビジネス制約事項

### 機能優先度制約
- **新機能開発**: パフォーマンス改善より優先
- **セキュリティ対応**: 最優先（パフォーマンス犠牲も許容）
- **法的要件**: GDPR対応でレスポンス増加許容

### ユーザー制約
- **レガシーブラウザ対応**: IE11サポート必須
- **モバイル環境**: 3G回線でも動作必要
- **アクセシビリティ**: WCAG 2.1 AA準拠必須

### 外部要因制約
- **パートナーAPI**: 変更不可・最適化不可
- **法的規制**: データ処理時間要件あり
- **契約条件**: SLA応答時間要件
```

## 6. テスト要件の仕様化

### 🧪 パフォーマンステスト要件の定義

パフォーマンステストを仕様書として文書化する方法：

#### 6.1 テスト分類と目的

| テスト種別 | 目的 | 実施タイミング | 合格基準 |
|-----------|------|---------------|----------|
| **性能テスト** | 正常負荷での性能確認 | 機能実装完了後 | 目標値達成 |
| **負荷テスト** | 高負荷時の性能劣化確認 | リリース前 | 許容値内での動作 |
| **ストレステスト** | 限界負荷での動作確認 | 四半期毎 | 適切な劣化・回復 |
| **エンデュランステスト** | 長時間運用での安定性 | 大型リリース前 | メモリリーク等なし |

#### 6.2 テストシナリオの定義

```markdown
## 負荷テストシナリオ例

### シナリオ1: 通常営業時間の負荷
- **実行時間**: 30分
- **同時ユーザー数**: 500人
- **操作パターン**: 
  - 商品閲覧: 70%
  - 検索: 20%
  - 購入: 10%
- **合格基準**: 
  - LCP P95 < 2.5秒
  - API応答時間 P95 < 300ms
  - エラー率 < 0.5%

### シナリオ2: ピーク時間の負荷
- **実行時間**: 15分
- **同時ユーザー数**: 1,000人
- **操作パターン**: 
  - 特定商品へのアクセス集中
- **合格基準**:
  - システムダウンしない
  - レスポンス時間 < 5秒
  - エラー率 < 5%
```

#### 6.3 テスト環境要件

```markdown
## テスト環境仕様

### 基本要件
- **本番環境と同一構成**: インフラ・ミドルウェア・設定
- **データ規模**: 本番の80%相当
- **ネットワーク**: 本番と同等の帯域・レイテンシ

### テストデータ要件
- **ユーザーデータ**: 10万件（個人情報は仮名化）
- **商品データ**: 1万件
- **取引履歴**: 100万件（過去1年分相当）

### 除外・簡略化項目
- **外部決済API**: モック化
- **メール送信**: ログ出力のみ
- **画像変換**: 固定画像使用
```

#### 6.4 テスト自動化要件

```markdown
## 自動化範囲

### 自動化対象
- **API性能テスト**: 全エンドポイント
- **フロントエンド性能テスト**: 主要ページ
- **定期回帰テスト**: 週1回実行

### 手動テスト
- **ユーザビリティ観点**: 体感速度確認
- **異常系**: 予期しない負荷パターン
- **新機能**: 初回性能テスト
```

## 7. 監視・アラート仕様の定義

### 📊 監視仕様の文書化

パフォーマンス監視を仕様として定義：

#### 7.1 監視対象の分類

```markdown
## 監視対象一覧

### レベル1: ビジネス影響度 高
| 監視項目 | 閾値 | アラート方法 | 対応SLA |
|---------|------|-------------|---------|
| **サイト全体可用性** | ダウン検知 | 即座電話 | 15分以内 |
| **決済API応答時間** | P95 > 3秒 | Slack + メール | 30分以内 |
| **ユーザー登録エラー率** | > 5% | Slack | 1時間以内 |

### レベル2: サービス品質
| 監視項目 | 閾値 | アラート方法 | 対応SLA |
|---------|------|-------------|---------|
| **ページ表示速度** | LCP P95 > 3秒 | Slack | 2時間以内 |
| **API全体エラー率** | > 1% | Slack | 2時間以内 |
| **データベース接続** | 失敗率 > 0.1% | メール | 4時間以内 |

### レベル3: 運用効率
| 監視項目 | 閾値 | アラート方法 | 対応SLA |
|---------|------|-------------|---------|
| **サーバーリソース** | CPU > 80% | ダッシュボード | 翌営業日 |
```
